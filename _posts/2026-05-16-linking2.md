# Linker가 하는 일
링커가 하는 일은 크게 두 가지가 있다.
1. Symbol resolution
2. Relocation

## 1. Symbol resolution
링커는 각각의 symbol reference를 정확히 하나의 symbol definition과 연결한다.  
쉽게 말해서,
``` markdown
main.o: sum 쓰고 싶음
sum.o: sum 여기 있음
링커: main.o의 sum 호출은 sum.o의 sum 함수로 연결
``` 
이것이 symbol resolution이다.  

다시 처음의 예시 코드를 보자.  
main.c
``` c
int sum(int *a, int n);
int array[2] = {1, 2};
int main(int argc, char** argv)
{
    int val = sum(array, 2);
    return val;
}
```
`main.c`에서 `array`, `main`은 symbol definition이고,  
`sum`은 symbol reference이다.  
실제 `sum` 함수의 몸체는 `sum.c`에 있기 때문이다.  

그리고 `sum.c`에서의 `sum`은 symbol definition이다.

즉, `main.c`의 sum reference가 `sum.c`의 sum definition으로 연결되는 것이다.  

링커는 reference 하나를 **정확히 하나**의 definition에 연결해야 한다.  
예를 들어 `main.c`에서 `sum()`을 호출했는데, `sum` 정의가 아예 없으면?  
당연히 에러가 난다.

반대로 `sum` 정의가 여러 개 있으면 어떻게 될까?  

## 4.2. Relocation
여러 오브젝트 파일에 따로 있던 코드 영억과 데이터 영역을 실행 파일 안에서 하나로 합치는 것이다.

예를 들어 컴파일 하면 이런 것들이 생긴다.
``` markdown
main.o
 ├─ .text   : main 함수 코드
 ├─ .data   : array 같은 초기화된 전역 변수
 └─ 기타 정보

sum.o
 ├─ .text   : sum 함수 코드
 └─ 기타 정보
 ```
 `text`에는 기계어 코드가, `.data`에는 초기값이 있는 전역/정적 변수 데이터가 들어가는 섹션이다.

 링커는 이것들을 최종 실행 파일 `prog` 안에서 합친다.  
 ``` markdown
 prog
 ├─ .text : main 함수 코드 + sum 함수 코드
 ├─ .data : array 데이터
 └─ 기타 정보
 ```

오브젝트 파일 `.o` 상태에서는 함수와 변수의 주소가 아직 최종 확정이 아니다.  
예를 들어 `main.o`에서는,  
```
main 함수: 이 파일의 .text 섹션 시작 기준 + 0번 위치
array: 이 파일의 .data 섹션 시작 기준 + 0번 위치
```
이런 식으로 아직 **"상대적인 위치"**만 알고 있는 것이다.  

그런데 실행 파일이 만들어질 때는, 상대적인 위치가 아닌 "최종 위치"가 정해진다.  

즉, relocation은:  
“오브젝트 파일 안에서 임시로 잡혀 있던 위치를, 실행 파일 기준의 최종 위치로 다시 배치하는 과정” 이다.
