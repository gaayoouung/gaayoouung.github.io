# 1. Linking이란?

여기 두 개의 c 파일이 있다.

main.c
``` c
int sum(int *a, int n);
int array[2] = {1, 2};
int main(int argc, char** argv)
{
    int val = sum(array, 2);
    return val;
}
```


sum.c
``` c
int sum(int *a, int n)
{
    int i, s = 0;
    for (i = 0; i < n; i++) {
        s += a[i];
    }
    return s;
}
```

main.c와 sum.c를 각각 컴파일한 뒤, 링킹 과정을 거쳐 실행 파일 prog를 만드는 명령어는 다음과 같다.  

``` bash
gcc -Og -o prog main.c sum.c
```
이 명령어는 겉으로 보면 한 번에 실행 파일 `prog`를 만드는 것처럼 보이지만, 내부적으로는 여러 단계를 거친다.  

``` markdown
main.c  →  main.o
sum.c   →  sum.o

main.o + sum.o → prog
```
각각의 소스파일이 먼저 따로따로 오브젝트 파일이 되고, 그 다음 링커가 그 오브젝트 파일들을 합쳐서 하나의 실행 파일을 만드는 것이다.  

## 1.1 오브젝트 파일이란?
`main.o`, `sum.o` 같은 파일을 object file이라고 한다.  
C 소스 파일을 기계어에 가까운 형태로 번역한 중간 결과물이다.  
아직 완전한 실행 파일은 아니다.  

Object file에는 세 종류가 있다.  
1. Relocatable object file (.o file)
여기서 `main.o`, `sum.o` 같은 파일들을 relocatable object file이라고 한다.
relocation에 대해서는 뒤에서 다시 설명한다.  

2. Executable object file 
마지막 결과물인 `prog`는 fully linked executable object file이다.  
파일 이름은 prog일 수도 있고, -o 옵션을 지정하지 않으면 기본적으로 a.out이라는 이름으로 만들어질 수 있다.  

완전히 실행 가능한 오브젝트 파일이라는 것이다.   
``` bash
./prog
```
요렇게 실행하면 됨  


3. Shared object file (.so file)
동적 링킹에서 사용되는 Relocatable file의 특별한 종류이다.  
실행 파일 안에 라이브러리 코드를 직접 넣는 대신, 실행 시점 또는 로딩 시점에 공유 라이브러리를 연결할 수 있게 해준다.  
지금은 자세히 다루지 않겠다.  

# 2. Linker를 사용하는 이유
이유는 크게 두 가지이다.
### 2.1. Modularity, 모듈성
프로그램을 하나의 거대한 파일로 만들면 너무 복잡하고 길어지기 때문에, 기능별로 파일을 나누는 것이다.
코드 관리가 쉬워지게 된다.
### 2.2. Efficiency, 효율성
separate compilation으로 시간 효율성을 늘릴 수 있다.
여러 개의 파일 중 특정 파일만 수정했다고 하자.
그러면 전체를 다시 컴파일 할 필요 없이, 수정한 파일만 컴파일 하면 된다.
큰 프로젝트에서는 파일이 수백, 수천 개일 수 있으니 매우 중요하다.

또한 여러 CPU 코어를 써서 여러 파일을 동시에 컴파일 할 수 있다.
그래서 빌드 시간이 줄어든다.

# 3. Linking의 두 가지 방식
## 3.1. Static Linking
정적 링킹은 실행 파일 안에 필요한 라이브러리 코드를 직접 포함하는 방식이다.  
예를 들어 프로그램이 어떤 라이브러리 함수 `foo()`를 쓴다면, 그 함수의 코드가 실행 파일 안에 들어가는 것이다.

장점은 실행 파일 하나만 있어도 돌아간다는 것이다.  
단점은 코드가 통으로 들어가기 때문에, 실행 파일 크기가 커질 수 있다.

여러 프로그램이 같은 라이브러리를 쓰면, 각 실행 파일마다 라이브러리 코드가 중복해서 들어가게 되는 것이다.

## 3.2. Dynamic Linking
동적 링킹은 실행 파일 안에 라이브러리 코드를 직접 넣지 않는다.  
대신 실행 파일에는 필요한 공유 라이브러리에 대한 정보만 들어가고, 실제 라이브러리 코드는 프로그램이 메모리에 로드될 때 또는 실행 중에 연결된다.    
예를 들어 리눅스에서의 `.so` 파일, 윈도우에서의 `.dll`파일이 이러한 동적 라이브러리이다.  

여러 프로그램이 같은 라이브러리를 쓰면, 메모리에 올라간 라이브러리 코드 하나를 같이 공유한다.  
따라서 메모리를 아낄 수 있다.  

정리하면 링킹은 여러 개의 오브젝트 파일과 라이브러리를 하나로 묶어 실행 가능한 파일을 만드는 과정이다.

이때 링커는 단순히 파일을 붙이기만 하는 것이 아니라, 각 파일에서 참조하는 함수나 전역변수의 위치를 찾아 연결하고, 코드와 데이터의 실제 주소를 조정한다.

이 과정에서 중요한 개념이 symbol resolution과 relocation이다.  
다음 글에서는 이 두 개념을 중심으로 링커가 실제로 무엇을 하는지 정리해보겠다.
