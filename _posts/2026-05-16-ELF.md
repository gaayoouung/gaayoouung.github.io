# Executable and Linkable Format (ELF)
오브젝트 파일과 실행 파일을 저장하는 표준 형식으로, 리눅스/유닉스 계열에서 사용하는 대표적인 바이너리 파일 형식이다.  

Relocatable object files (.o)  
Executable object files    
Shared object files (.so)  
-> 모두 겉으로는 역할이 다르지만, 파일 형식은 같은 ELF 계열이다.  
``` text
main.c
↓
main.o    ← relocatable object file, ELF

sum.c
↓
sum.o     ← relocatable object file, ELF

main.o + sum.o
↓ 링커가 symbol resolution + relocation 수행
prog      ← executable object file, ELF
```

## ELF Object File Format 

ELF 파일은 이런 식으로 구성되어 있다.  

| ELF 구조 |
| ------- |
| ELF header |
| Segment(Program) header table |
| .text |
| .rodata |
| .data |
| .bss |
| .symtab |
| .rel.text (.rela.text) |
| .rel.data |
| .debug |
| Section header table |  

하나씩 무엇이 들어 있는지 보자.  

### ELF header
ELF 파일 맨 앞에 있는 기본 정보이다.  
Word size, byte ordering, file type, machine type 등의 내용이 포함된다.  
ELF 파일 자체의 신분증 같은 느낌이다.  

예를 들어,  
- 이 파일이 32비트인지 64비트인지
- 리틀 엔디안인지 빅 엔디안인지
- .o 파일인지 .so 파일인지
- 어떤 CPU 아키텍처용인지  
이러한 정보를 담고 있다.

### Segment(Program) header table (required for executable)
실행 파일을 **메모리에 올릴 때** 필요한 정보이다.  
각 entry는 로더가 바라보는 segment 정보를 담고 있다.  
Page size, virtual address memory segments, segment sizes 등의 내용이 포함된다.  

프로그램이 실행되려면 파일 내용을 메모리에 배치해야 하는데, 그때 운영체제 로더가 segment header table을 보고:
- 어느 부분을 메모리 어디에 올릴지
- 읽기/쓰기/실행 권한은 어떻게 줄지
- 각 segment 크기는 얼마인지  
등을 결정한다.  

### .text section
기계어 코드가 들어가는 section이다.  
C 코드로 보면 함수들이 컴파일된 결과가 들어가는 것이다.  
함수들이 컴파일되어 CPU가 실행할 수 있는 명령어가 되고, 그 명령어들이 .text section에 들어가는 것이다.  
즉, .text = 실행할 코드 이다.  

### .rodata section
rodata는 read-only data의 줄임말이다.  
즉, 읽기만 가능한 데이터가 들어가는 것이다.  
대표적으로 문자열 상수 같은 것들이 들어간다.  

예를 들어 `printf("hello\n");`에서 "hello\n" 같은 문자열 리터럴이 저장된다.  

또는 switch문 최적화에 쓰이는 jump table 같은 것도 들어갈 수 있다.  

### .data section
**초기값이 있는** 전역 변수 또는 static 변수가 들어가는 section이다.  
여기서 static 변수는 함수 내부의 static local variable도 포함된다.  

```c
int x = 10;
int array[2] = {1, 2};

void f(void)
{
    static int count = 5;
}
```   

이렇게 처음부터 값이 정해져 있는 변수들이 들어간다.  

### .bss section
**초기화되지 않은** 전역 변수 또는 static 변수가 들어가는 section이다.  
```c
int x;
static int count;
int arr[1000];
```

C에서는 전역 변수나 static 변수에 초기값을 주지 않으면 자동으로 0으로 초기화 된다.  

예를 들어, `int arr[1000000];` 이 있다고 하자.  
이 배열은 전부 0으로 초기화 될 예정이지만, 실행 파일에 0을 백만 개 저장하면 공간 낭비이다.  
그래서 이러한 경우 `.bss`는 보통 "이만큼의 크기의 0으로 초기화될 공간이 필요하다" 라는 정보만 저장하고, 실제 0 데이터는 넣지 않는다.  

### .symtab section
함수 이름, 전역 변수 이름, static 변수 이름, section 이름, 그리고 각 symbol이 어느 section의 어느 offset에 있는지 같은 정보가 들어간다.  
일반 지역변수는 보통 링커가 사용하는 symbol table의 주요 대상이 아니지만, static local variable은 저장 공간이 고정적으로 필요하기 때문에 local symbol로 나타날 수 있다.  

예를 들어,
``` 
main
sum
array
.text
.data
.bss
``` 
이러한 것들이 들어갈 수 있다.  

링커가 symbol resolution을 할 때 이 정보를 본다.  

### .rel.text section
`.text` section에 대한 relocation 정보이다.  
위에서 `.text`에는 기계어 코드가 들어간다고 했었다.  
따라서 `.rel.text`는 "코드 영역 안에서 나중에 주소를 고쳐야 하는 위치 목록"이다.  

예를 들어 `main.c`의 `int val = sum(array, 2);`라는 코드를 보자.  
컴파일해서 `main.o`를 만들 때는 아직 `sum` 함수의 최종 주소를 모른다.  

따라서 `main.o`의 `.text` section에는 대충 이런 코드가 들어갈 수 있다.  
`call sum`  
그런데 이 `call` 명령어의 실제 목적지 주소는 아직 확정되지 않았다.  

그래서 `.rel.text`에 이러한 정보가 기록된다. 
- .text section의 몇 번째 바이트에 있는 call 명령어는 나중에 sum의 실제 주소가 정해지면 수정되어야 함

링커는 나중에 이 `.rel.text` 정보를 보고 `call` 명령어의 주소 부분을 고치게 되는 것이다.  

### .rel.data section
`.data` section에 대한 relocation 정보이다.  
"데이터 영역 안에서 나중에 주소를 고쳐야 하는 포인터 정보" 이다.  

예를 들어 이런 코드가 있다.  
```c
int x = 10;
int *p = &x;
```
`p`는 전역 포인터 변수고, 초기값은 `x`의 주소이다.  
그런데 `.o` 파일을 만들 때는 `x`가 최종 실행 파일에서 몇 번째 주소에 놓일지 아직 모른다.  
따라서 `p` 안에 넣을 실제 주소도 아직 모른다.  

그러면 `.rel.data`에 이러한 정보가 들어간다.  
- .data section 안의 p 위치는 나중에 x의 최종 주소로 수정해야 함.  

| section     | 무엇을 고치나   | 예시                           |
| ----------- | --------- | ---------------------------- |
| `.rel.text` | 코드 안의 주소  | `call sum`, 전역 변수 접근 명령어     |
| `.rel.data` | 데이터 안의 주소 | `int *p = &x;` 같은 전역 포인터 초기값 |

### .debug section
디버깅 정보가 들어가는 section이다.  

예를 들어 컴파일할 때,  
`gcc -g main.c sum.c`처럼 `-g` 옵션을 주면 디버거가 사용할 수 있는 정보가 들어간다.  

대표적으로 :
- C 소스 코드의 몇 번째 줄이 어떤 기계어 명령어와 대응되는지
- 변수 이름, 함수 이름, 타입 정보가 무엇인지  
이러한 정보들이다.  

### Section header table
ELF 파일 안에 있는 각 section의 위치와 크기를 알려주는 표이다.  
실행 자체에는 program header table이 더 중요하고, section header table은 링커나 디버거가 ELF 내부 section을 분석할 때 중요하다.  

ELF 파일 안에는 여러 section들이 쭉 들어 있다.  
링커나 디버거가 이 section들을 찾으려면, 각각이 어디에 있는지를 알아야 한다.  
이러한 정보를 담은 게 section header table이다.  

각 section의 목차 같은 느낌이다.  

**Segment header table과 section header table 구분!**  
Section  
→ 링커 관점  
→ .text, .data, .bss, .symtab처럼 파일 내부를 세부적으로 나눈 단위

Segment  
→ 실행/로딩 관점  
→ 운영체제가 실행 파일을 메모리에 올릴 때 쓰는 큰 단위

## ELF 파일 전체 한 번에 정리
ELF header  
→ 파일 기본 정보: 32/64비트, 엔디언, 파일 종류, CPU 종류

Segment(Program) header table  
→ 실행 시 메모리에 어떻게 올릴지에 대한 정보

.text  
→ 실행할 기계어 코드

.rodata  
→ 읽기 전용 데이터, 문자열 상수 등

.data  
→ 초기값이 있는 전역/static 변수

.bss  
→ 초기화 안 된 전역/static 변수, 실행 시 0으로 채워짐

.symtab  
→ 함수/변수/section 이름과 위치 정보

.rel.text / .rela.text 
→ 코드 영역에서 나중에 주소 수정해야 할 정보

.rel.data / .rela.data 
→ 데이터 영역에서 나중에 주소 수정해야 할 정보

.debug  
→ 디버깅 정보, gcc -g 옵션 사용 시 생성

Section header table  
→ 각 section의 위치와 크기를 알려주는 표
