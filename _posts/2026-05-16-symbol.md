# Symbol이란?
링커 관점에서 symbol은 오브젝트 파일 안에서 이름으로 식별되는 함수, 전역 변수, static 변수, section 등을 말한다.  
대표적으로 함수 이름과 전역 변수 이름이 중요하다.  

아래 코드에서 심볼을 찾아보자.  
``` c
int sum(int *a, int n);
int array[2] = {1, 2};
int main(int argc, char** argv)
``` 
여기서 symbol은:
sum, array, main이다.  

어셈블러가 오브젝트 파일을 만들 때, 이러한 symbol 정보를 **symbol table**에 저장한다.  
`main.c`를 컴파일하면 `main.o`가 생기는데, 그 안에는 기계어 코드만 있는 것이 아니라 이러한 symbol 정보들도 들어 있는 것이다.
``` text
symbol 이름
symbol 크기
symbol 위치
symbol 종류
```

대충 이런 식으로 정보가 들어갈 수 있다.
``` text
main    함수    이 파일 안에 정의됨
array   전역변수 이 파일 안에 정의됨
sum     함수    이 파일에서 사용하지만 정의는 없음  
```  

링커가 보는 symbol은 크게 3종류이다.  
1. Global symbols
2. External symbols
3. Local symbols  

## 1. Global symbols
> 어떤 모듈에서 정의되었고, 다른 모듈에서도 참조할 수 있는 symbol  

(여기서 모듈은 보통 각각의 소스 파일이 컴파일되어 만들어진 오브젝트 파일을 의미)  

대표적으로 'non-static C functions', 'non-static global variables'   
```c
int x = 10;

void foo() {
}
```
여기서 `x`와 `foo`는 global symbol이다.  
`static`이 안 붙었고, 바깥 파일에서도 접근이 가능하다.  

다른 파일에서 이런 식으로 쓸 수 있다.  
```c
extern int x;
void foo();

int main() {
    foo();
    return x;
}
```
## 2. External symbols
> 현재 파일에서 사용하고 있지만, 실제 정의는 다른 파일에 있는 global symbol

main.c : 
```c
int sum(int *a, int n);

int array[2] = {1, 2};

int main(int argc, char **argv)
{
    int val = sum(array, 2);
    return val;
}
```

sum.c : 
```c
int sum(int *a, int n)
{
    int i, s = 0;

    for (i = 0; i < n; i++) {
        s += a[i];
    }

    return s;
}
```
`main.c` 입장에서 보면, `sum`은 external symbol이다.  

## 3. Local symbols
> 한 오브젝트 파일 안에서만 보이는 symbol

C에서 `static`을 붙인 전역 변수나 함수가 여기에 해당된다.  
```c
static int count = 0;
static void p() {}
```
symbol table에는 들어가지만, 외부에 공개되는 global symbol이 아니라 local linker symbol로 들어간다.  

Linker의 local symbol과 C 코드의 지역변수는 다르다.  
예를 들어,  
```c
int main() {
    int val = 10;
    return val;
}
```
여기서 `val`은 C 언어에서 지역 변수이다.  
하지만 Linker에서 `val`과 같은 지역 변수는 보통 symbol로 다루지 않는다.  
일반적인 지역변수는 함수 실행 중 스택 프레임에 생겼다가 함수가 끝나면 사라지기 때문이다.  

### 그렇다면 **"local static variable"**은 어떻게 될까?  
일반 지역변수와 달리 local static variable은 보통 symbol로 다뤄질 수 있다.   
```c
int main() {
    static int val = 10;
    return val;
}
```
local static은 함수 안에 선언되어 있지만, 스택에 생기지 않고 프로그램이 종료될 때까지 계속 살아있다.  
(local 성질이 사라지는 것은 아니다. 여전히 외부에서 볼 수 없다.)  
따라서 메모리 어딘가에 고정된 저장 공간이 필요하다.  
그래서 보통 메모리의 `.data` 또는 `.bss` 섹션에 저장된다.  
그 위치를 표현하기 위해 컴파일러가 내부적으로 symbol을 만들 수 있다.  

### static local variable끼리 서로 같은 이름을 가지게 되면 어떻게 될까?  
```c
int f()
{
    static int x = 0;
    return x;
}

int g()
{
    static int x = 1;
    return x;
}
```
여기서 `f`와 `g` 안의 `x`는 이름은 같지만 서로 다른 변수이다.  
둘 다 static local variable이므로 stack이 아니라 `.data`나 `.bss`에 공간이 잡힌다.  
링커 입장에서는 둘 다 symbol table에 local symbol로 들어가게 되는데, 이름이 같은 `x`이면 충돌할 수 있다.  

그래서 컴파일러가 내부적으로 "unique name"을 만들어준다.  
사람이 보는 C 코드에서는 둘 다 `x`지만, 오브젝트 파일 내부에서는 구분되는 이름으로 저장되는 것이다.  

### symbol 찾기 예시
다음 C 파일에서 symbol을 찾아보자.  
```c
#include <stdio.h>

int incr = 1;

static int foo(int a) {
    int b = a + incr;
    return b;
}

int main(int argc, char* argv[]) {
    printf("%d\n", foo(5));
    return 0;
}
```
여기서 symbol은 `incr`, `foo`, `main`, `printf`이다.  
1. `incr`
: `static`이 없는 전역 변수이므로 global symbol
2. `foo`
: static이 있으므로 외부 파일에서는 참조할 수 없는 local symbol
3. `main`
: static이 없는 전역 함수이므로 global symbol
4. `printf`
이 파일에서 호출하지만, 실제 정의는 C 표준 라이브러리에 있는 external symbol  

# 중복 symbol 정의 문제
같은 이름의 전역 symbol이 여러 파일에 있을 때 어떻게 될까?  
symbol은 strong symbol과 weak symbol로 나뉜다.  

## Strong symbol
- 전역 함수
- 초기화된 전역 변수
```c
int foo = 5;
void p1() {}
```

## Weak symbol
- 초기화되지 않은 전역 변수
- extern 선언
```c
int foo;
extern int foo;   // "foo"가 어딘가에 있다"는 선언이므로 정의가 아니라 참조에 가까움
```

## Rule 1. Multiple strong symbols are not allowed
> 같은 이름의 strong symbol이 있으면 링커 에러

`a.c` : `int x = 1;`   
`b.c` : `int x = 2;`  
-> 둘 다 초기화된 전역 변수 이므로 strong symbol이다.  
-> link-time error  

함수의 경우도 마찬가지이다.  

## Rule 2. strong 하나 + weak 여러 개 -> strong 선택
> 같은 이름에 대해 strong symbol이 하나 있고, weak symbol이 여러 개 있으면, 링커는 strong symbol을 선택한다.  

`p1.c` : `int foo = 3;`  
`p2.c` : `int foo;`  
-> 링커는 strong인 `int foo = 3;`를 진짜 `foo`로 선택한다.  
-> 모든 `foo` 참조는 같은 하나의 `foo`를 가리킴  

## Rule 3. weak만 여러 개면 아무거나 선택  
> 같은 이름의 weak symbol만 여러 개 있으면, 링커가 임의로 하나를 선택한다.  

`a.c` : `int x;`   
`b.c` : `int x;`  
둘이 따로따로 `x`가 생기는 게 아니라, 하나의 같은 `x`를 공유하게 된다.  
(전통적인 CS:APP 기준이다. 요즘 GCC는 기본 설정 변화 때문에 이 규칙이 에러가 될 수도 있다.)  

## Linker는 타입 검사를 하지 않는다.  
그냥 symbol 이름이 같으면 같은 이름으로 보고 연결하게 되므로, 위험한 문제가 생길 수 있다.  

다음 두 개의 파일이 있다고 하자.  
```c
int x;
int y;
p1() {}
```
```c
double x;
p2() {}
```
위의 `x`와 아래의 `x` 모두 초기화 되지 않은 전역 변수이므로 weak symbol이다.  
링커는 이름만 보고 둘을 같은 `x`로 합쳐버릴 수 있다.  

그런데 타입 크기가 4바이트, 8바이트로 각각 다르다.  
아래쪽 파일에서는 `x`를 double이라 생각하고 8바이트를 쓴다.  

하지만 실제로 링커가 위쪽의 `int x`를 기준으로 잡으면,  
아래쪽에서 `double x`라고 생각하고 8바이트를 쓰는 순간,  
x 4바이트 + 그 다음 y 4바이트 까지 덮어쓸 수 있다.  

링커가 타입 체크를 하지 않기 때문이다.  

다음 예시를 보자.  
```c
int x = 7;
int y = 5;
p1() { }
```
```c
double x;
p2() { }
```
이번에는 위쪽 파일의 `x`와 `y`가 초기화되어 있으므로 strong symbol이다.  
아래의 `x`는 초기화되지 않은 전역 변수이므로 weak symbol이다.  

strong과 weak가 같이 있으므로 strong이 선택된다.  
즉 최종 `x`는 위쪽의 `int x = 7`이다.  

하지만 아래 파일에서 여전히 `x`를 double이라고 생각하고 사용할 수 있다.  
위의 예시처럼 8바이트를 쓰려고 하게 되고, 옆에 있던 y까지 덮어쓸 수 있게 된다.  

타입 크기가 같아도 이상한 결과가 나올 수 있다.  
다음 두 개의 파일을 있다.  
```c
long int x;   

int main(int argc, char *argv[]) {
    printf("%ld\n", x);
    return 0;
}
```
여기서 `x`는 초기화되지 않은 전역 변수이므로 weak symbol이고, long은 8바이트이다.  

```c
double x = 3.14
```
여기서 `x`는 초기화된 전역 변수이므로 strong symbol이고, double 또한 8바이트이다.  

규칙에 의해 strong symbol인 `double x`가 선택된다.  

하지만 위쪽 파일에서는 그 같은 메모리를 `long int x`라고 생각하고 읽는다.  

저장은 `double x = 3.14`로 되지만, 위쪽 파일에서 읽을 떼는 `long int x`로 읽는 것이다.  
즉, double 3.14의 비트 패턴을 그대로 long int로 해석해서 출력하게 된다.  

따라서 아래쪽 파일에서 double을 기준으로 저장한 x를 위쪽 파일에서 long을 기준으로 비트 패턴을 해석하므로 이상한 정수가 출력될 수 있는 것이다.    

각 `.c` 파일은 따로 컴파일 되고, 링커는 타입 체크를 하지 않기 때문에 이런 문제가 발생할 수 있는 것이다.  

## 이런 문제를 최소화하는 방법
1. 무분별한 전역 변수 사용은 피해라.  
2. 다른 파일에서 쓸 필요가 없는 전역 변수라면 `static`을 붙여라.  
3. 전역 변수를 정의할 거면 초기값을 줘라.  
4. 다른 파일에 정의된 전역 변수를 가져다 쓸 때는 `extern`을 써라.  
: `int g;` -> `extern int g;`
: 함수의 경우에는 extern을 안 붙여도 됨.
: `int f()` 는 몸체가 없으니 정의가 아니고 선언이다. -> strong symbol definition 안 생김  

다음 글에서는 본격적으로 Linker가 하는 두 가지 일인 "symbol resolution"과 "relocation"에 대해 다루겠다.
